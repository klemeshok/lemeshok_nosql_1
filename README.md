## 3. Відповіді на теоретичні питання з частин 1–3

### 1.1. Чому аудіо-характеристики винесені в окремий об’єкт audio_features, а не зберігаються плоско? Коли таке вкладення вигідне, а коли створює проблеми?

Винесення аудіо-характеристик в audio_features логічно групує пов'язані дані. Трек має метадані (назва, виконавець, жанр) і технічні акустичні метрики. Групування робить документ чистішим і зручнішим для читання. Окрім того, подібний підхід дозволяє спростити код застосунку (можна отримати або передати весь об'єкт `audio_features` одразу), а кореневий рівень документа залишається незахаращеним.

Однак якщо вкладеність занадто глибока, синтаксис запитів стає більш громіздким. Наприклад, в нашому випадку другий рівень вкладеності `"audio_features.tempo"` - це трохи довше, аніж перший рівень `"tempo"`. А на 3-му, 4-му і подальших рівнях стає зовсім незручно, і це гальмує розробку.

До цього додається ризик випадкового перезапису. Якщо виконати оновлення вигляду `db.tracks.updateOne(..., { $set: { audio_features: { tempo: 120 } } })`, MongoDB повністю перезапише об'єкт audio_features, видаливши всі інші метрики. Правильний запит вимагає конструкції `{ $set: { "audio_features.tempo": 120 } }`. Тобто при оновленні вкладених полів необхідно суворо використовувати dot notation.

### 1.2. Чому виконавці зберігаються як масив, а не як рядок? Які запити стають простішими?

Зберігання виконавців як масиву дозволяє MongoDB створити індекс по виконавцям, а також робити прості запити по окремим виконавцям. Наприклад, якби ми залишили рядки замість масивів, то рядок "Gen Hoshino; Someone Else" для пошуку "Gen Hoshino" вимагав би регулярного виразу, що повільно і дорого у порівнянні з `db.tracks.find({ artists: "Gen Hoshino" })`. Підхід з масивами також дозволяє використовувати масив-специфічні оператори: `$in`, `$all` та `$unwind`, що зручно для аналітики.

### 1.3. Що таке $out і чим він відрізняється від $merge? Коли використовувати кожен?

`$out`: Бере результати пайплайну та записує їх у колекцію. При цьому він повністю замінює цільову колекцію, якщо вона вже існує. Найкраще підходить для повної перебудови даних або створення нових матеріалізованих представлень з нуля.

`$merge`: Він теж записує результати в колекцію, але може робити це більш "точково": вставляти, оновлювати або замінювати існуючі документи. Крім того, він не видаляє вже існуючу колекцію. Найкраще підходить для інкрементальних оновлень (наприклад, додавання даних за певний день до існуючої аналітичної таблиці).

### 2.1. Для чого використовується інструкція $unwind?

Оператор `$unwind` використовується для "розгортання" масивів. Якщо у нас є документ з масивом із 3 виконавців, `$unwind` створить 3 окремих документи (повні копії оригінального), але в кожному з них масив буде замінено на один конкретний елемент з цього масиву. Це необхідно, коли нам треба згрупувати дані або порахувати статистику не по документу в цілому, а по кожному елементу масиву окремо (як ми це робили з пошуком популярних артистів). Без `$unwind` база даних намагалася б згрупувати треки за всіма масивами виконавців цілком, а не за індивідуальними іменами.

### 2.2. Чим $stdDevPop відрізняється від $stdDevSamp?

Перший оператор рахує стандартне відхилення генеральної сукупності. Другий - стандартне відхилення вибірки. На практиці `$stdDevPop` використовується, коли наші дані — це і є вся наявна генеральна сукупність. Формула ділить дисперсію на загальну кількість елементів N. `$stdDevSamp` використовується, коли наші дані — лише невелика вибірка з більшого набору, і ми хочемо оцінити відхилення для всієї цієї більшої невідомої сукупності. Тут використовується поправка Бесселя (ділення на N - 1), що робить результат трохи більшим, щоб компенсувати похибку неповної вибірки.

У нашому випадку ми аналізуємо конкретний скачаний датасет Spotify як повний набір даних, тому логічно використовувати саме `$stdDevPop`.

### 3.1. У запиті 1 ми фільтруємо виконавців, у яких менше 5 треків. Як зміниться результат, якщо знизити поріг до 1? А що станеться, якщо вибирати виконавців із більш ніж 50 треками? Поясніть результат.

Якщо знизити поріг до 1: У топ вирвуться виконавці, у яких в датасеті є всього один трек, але з дуже високою популярністю. Зокрема, перші 2 позиції зміняться на `{ artist: 'Quevedo', avg_popularity: 99 }`, `{ artist: 'Bizarrap', avg_popularity: 99 }`. А лідер рейтингу для виконавців з 5+ треками `{ artist: 'Olivia Rodrigo', avg_popularity: 87.4 }` опуститься на 8 місце. З точки зору аналітики поріг в 1 трек створює статистичний шум, який ховає справді успішних виконавців.

Якщо підвищити поріг до 50: Ми відсіємо багато популярних артистів із середньою дискографією. У топі залишаться лише "ветерани" індустрії або мега-поп-зірки, у яких завантажено величезну кількість матеріалу. Зокрема, перші 2 позиції займуть: `{ artist: 'The Neighbourhood', avg_popularity: 75.6 }` та `{ artist: 'Halsey', avg_popularity: 71.5 }`. Як бачимо, при цьому середня популярність топових виконавців нижча, ніж при ліміті в 5 і тим більше в 1 трек, оскільки тримати стабільно хітовий рівень на 50 треках значно складніше, ніж випустити 5 успішних синглів.

### 3.2. У запиті 3 ми фільтруємо жанри з менше ніж 100 треками. Чи зміниться результат, якщо знизити поріг до 50? Поясніть результат.

На практиці результат не змінився. І це легко пояснити, оскільки обидва пороги повертають 114 жанрів, тобто один і той самий їх набір. Але навіть якби це було не так, 50 треків — це все ще достатньо високий рівень статистичної значущості для формування стабільного середнього значення (у класичній статистиці вибірки від 30 одиниць вже дозволяють згладжувати аномалії).

Проте, стовідсоткова ідентичність результатів при зміні порогів ніколи не гарантується. У нашому випадку відбувається двократне зниження ліміту (зі 100 до 50). Коли ми працюємо з вибірками, що вимірюються лише десятками зразків, кожен окремий "екстремальний" трек починає мати більшу математичну вагу. Таке різке зниження порогу цілком могло привести до того, що у фінальний рейтинг протиснувся б якийсь мікро-жанр, хоча загальна картина залишиться все одно стабільною.

## 4. Частина 4 — аналіз індексів

### 4.1. Аналіз запиту та індексація

#### Логи виконання команди explain("executionStats")

**До створення індексу:**

```json
{
  "queryPlanner": {
    "winningPlan": {
      "stage": "SORT",
      "inputStage": {
        "stage": "COLLSCAN"
      }
    }
  },
  "executionStats": {
    "executionTimeMillis": 76,
    "totalKeysExamined": 0,
    "totalDocsExamined": 113999
  }
}

```

**Після створення індексу (`idx_genre_popularity_danceability`):**

```json
{
  "queryPlanner": {
    "winningPlan": {
      "stage": "FETCH",
      "inputStage": {
        "stage": "IXSCAN",
        "indexName": "idx_genre_popularity_danceability"
      }
    }
  },
  "executionStats": {
    "executionTimeMillis": 2,
    "totalKeysExamined": 412,
    "totalDocsExamined": 354
  }
}

```

#### Що змінилося в плані виконання?

Як видно з наведених вище логів, план виконання завдяки індексу кардинально оптимізувався.

Зокрема, до індексації база даних виконувала повне сканування колекції (stage: `"COLLSCAN"`), а потім змушена була сортувати результати в пам'яті (stage: `"SORT"`). Після створення індексу база одразу сканує B-Tree структуру індексу (stage: `"IXSCAN"`) і витягує готові документи (stage: `"FETCH"`). Етап ручного сортування повністю зник.

Це призвело до кардинального зменшення обсягу роботи: У першому лозі `totalDocsExamined` дорівнює 113999, тобто база прочитала всі існуючі записи. У другому лозі це число впало до 354, що дорівнює кількості повернутих результатів (`nReturned`). Як наслідок, майже в 40 разів зменшився час виконання (`executionTimeMillis`) — із 76 мс до 2 мс.

#### Як зрозуміти, що індекс використовується? Наведіть скріншот або значення полів із explain(), які це підтверджують.

Про використання створеного індексу найяскравіше свідчать наступні поля з логу ПІСЛЯ створення індексу:

* Поле `queryPlanner.winningPlan.inputStage.stage` змінилося на `"IXSCAN"` (Index Scan).
* Поле `queryPlanner.winningPlan.inputStage.indexName` містить точну назву нашого індексу: `"idx_genre_popularity_danceability"`.
* Поле `executionStats.totalKeysExamined` дорівнює 412. Це доводить, що рушій бази даних сканував ключі індексу для пошуку записів, тоді як у запиті без індексу цей показник дорівнював нулю.

### 4.3. Покривний запит

Припустимо, що індекс із завдання 1 вже існує.

Дано запит:

```javascript
db.tracks.find({
  track_genre: "pop",
  popularity: { $gte: 70 }
});

```

**Питання: Чи є цей запит покривним (covered query)? Надайте розгорнуту та обґрунтовану відповідь**

Замість того, щоб просто теоретизувати, перевіримо це на практиці за допомогою методу `explain`:

```javascript
db.tracks.find({
  track_genre: "pop",
  popularity: { $gte: 70 }
}).explain("executionStats");

```

Отримаємо (ключові метрики):

```json
{
  "queryPlanner": {
    "winningPlan": {
      "stage": "FETCH",
      "inputStage": {
        "stage": "IXSCAN",
        "indexName": "idx_genre_popularity_danceability"
      }
    }
  },
  "executionStats": {
    "totalKeysExamined": 317,
    "totalDocsExamined": 317
  }
}

```

Тож бачимо, що досліджуємий нами запит **не є покривним**.

Щоб запит вважався покривним, абсолютно всі поля, які він повертає, мають знаходитися безпосередньо в самому індексі. У такому випадку базі даних взагалі не доводиться звертатися до оригінальної колекції — вона віддає результат виключно на основі індексу. Наведені вище логи виконання чітко доводять, що це не так, тому що:

* У полі `queryPlanner.winningPlan.stage` ми бачимо етап `"FETCH"`. Це означає, що після швидкого знаходження потрібних записів у B-Tree дереві індексу (`"IXSCAN"`), база даних змушена фізично звертатися до колекції, щоб "витягти" повні документи.
* Поле `executionStats.totalDocsExamined` показує `317`. Якби запит був повністю покривним, базі не довелося б читати самі документи, і це значення дорівнювало б `0` (при цьому `totalKeysExamined` все одно залишалося б `317`).

Запит не став покривним тому, що метод `find()` за замовчуванням повертає весь документ цілком (разом з `_id`, виконавцями, назвою треку тощо). Оскільки цих полів в індексі `idx_genre_popularity_danceability` немає, MongoDB змушена завантажувати їх з диска. Щоб перетворити запит на дійсно покривний, необхідно додати проєкцію, яка відріже поле `_id` і залишить лише ті поля, які вже лежать в індексі:

```javascript
db.tracks.find(
  { track_genre: "pop", popularity: { $gte: 70 } }, 
  { _id: 0, track_genre: 1, popularity: 1 }
);

```
